# 先和运营商议

本次大概有多少个商品、多少个库存、活动时间、优惠力度等，确定大概有多大的访问量

# 设置REDIS商品库存KEY

在活动开始前半个小时，由运营手动去设置 商品的库存KEY


# 架构

DNS->LVS->网关-》GO集群-》REDIS集群

# 热KEY

单KEY，访问量大，容易打挂了。HASH分散KEY
比如：
- 商品库存为100
- key name=sec_kill:food_id:1000
- 根据UID最后一位，设置成10个新KEY
- sec_kill:food_id:1000:1.....sec_kill:food_id:1000:10

# 抢购URL动态生成

抢购的后端API的URL，不要提前露出去，避免外挂提前写脚本抢购

活动开始前一分钟：请求后端接口，后端返回：一个URL
比如：www.baidu.com/sec_skill/$动态参数，每个用户不一样/
动态参数：md5（密钥+UID）


在即将开始抢购的1分钟前，去请求CDN,拿到动态URL


# 网关限流

UID：每秒一次
IP：这个值得大一些，很可能一个IP有多个用户同时在抢
Global QPS：压测系统最大能抗多少  X 80%

# 后端逻辑

判断：URL中的参数，正确与否

get redis:    user_buy:food_id:1
判断：KEY存在，判断value 的status，返回给前端

判断：GO进程内存中 是否有标识：卖光了

使用LUA脚本：
- get key   ,是否存在
- key <= 0 ,卖光了，GO缓存到进程内存中
- 执行 -1  decr
 ----->lua 结束
 
 如果成功：写新KEY：user_buy:food_id:1 => timestamp: 状态(已抢到，等后端DB业务完成)
如果失败：redis - 1 失败，可能有并发的写入

给MQ发消息
